Components

The new FCDO Design System will introduce a structured approach to designing and creating components, ensuring consistency, accessibility, and efficiency across all digital services.

Component design process

New components in the FCDO Design System will follow a structured lifecycle, including:

Research and discovery

Identifying user needs, accessibility requirements, and ensuring alignment with existing GOV.UK standards.

Wireframing and prototyping

Creating low-fidelity and high-fidelity designs to visualize the component's look and behavior.

Testing and accessibility reviews

Ensuring compliance with WCAG 2.2 accessibility standards and testing with real users.

Development and documentation

Writing scalable, reusable CSS & JavaScript, with detailed documentation in the FCDO Design System.

Iterative updates

Based on user feedback, data insights, and emerging best practices.

Component structure

Each component will have:

HTML Structure

Identifying user needs, accessibility requirements, and ensuring alignment with existing GOV.UK standards.

CSS Styling

Using FCDO specific styling (e.g., fcdo-color-primary, fcdo-spacing-m) layered on top of GOV.UK styles.

JavaScript (if needed)

Lightweight, progressive enhancement-based functionality for interactive elements.

New and enhanced UI components

The system will introduce new and customised versions of common GOV.UK components, including:

  • Header
  • Navigation Bar
  • Sub Navigation
  • Counter
  • Progress Bar
  • Sortable Table and Filter
  • Add Another Item
  • Date Picker
  • Notification Badge

Integrating Nunjucks

The new FCDO Design System will eventually use Nunjucks as a templating engine to streamline the integration of reusable components. This ensures consistency across services while allowing flexibility in implementation.

How Nunjucks Will Be Used in the FCDO Design System

Reusable Component Macros

Each component will have a Nunjucks macro, allowing developers to include standardized elements without manually writing repetitive HTML.

Example: A button component can be called using a simple macro:

            
from "fcdo/components/button/macro.njk" import fcdoButton

fcdoButton({
    text: "Submit",
    href: "/submit",
    type: "primary"
})
Component Configuration with Parameters

Each macro will accept parameters to customize attributes like text, styles, and behavior while maintaining consistency.

Example: A notification banner macro might accept parameters like type (success, warning, error) and message content.

            
fcdoNotificationBanner({
    type: "success",
    text: "Your application has been submitted successfully."
})
Inheritance and Layouts

Nunjucks supports template inheritance, meaning that core layouts (e.g., a standard FCDO page structure) can be defined in a base template and extended across multiple pages.

            
{ extends "fcdo/layouts/main.njk" }
  
{block pageTitle}  Home - FCDO   {endblock}
  
{block content}
      # content here
{endblock}
Dynamic Data Integration

Nunjucks templates can dynamically pull data from backend services, APIs, or content management systems to populate UI components.

Example: Rendering a list of services dynamically:

            
{ for service in services }
    #html code here
{endfor}
Extending GOV.UK Components

Since the FCDO Design System builds on the GOV.UK Design System, existing GOV.UK components can be extended and customized within Nunjucks templates.

Example: Customizing the GOV.UK details component with FCDO styles:

            
{ from "govuk/components/details/macro.njk" import govukDetails }
  
govukDetails({
    classes: "fcdo-details",
    summaryText: "More information",
    text: "This is additional content customized for FCDO."
})

Benefits of using Nunjucks

Reusability

Reduces duplication by using macros for commonly used components.

Consistency

Ensures all services follow the same design guidelines.

Efficiency

Speeds up development by providing pre-built templates.

Flexibility

Allows services to customize components while maintaining core design standards.
